# Запуск и взаимодействие

## liveTop

Запуск сервиса liveTop и вспомогательного брокера:

```bash
docker compose up --build
```

Пример запроса к запущенному сервису liveTop:

```bash
docker compose exec live_top curl http://localhost:8080/?N=10
```

(или любой другой на ваше усмотрение HTTP GET с параметром N)

Тесты liveTop:

```bash
cd liveTop && go test ./...
```

## broker

Брокер с момента запуска работает детерменированно, отправляя события, прописанные в [broker/events.json](broker/events.json). События содержат поля: query (отправится по NATS), delta (отправится по NATS), start (секунда после запуска брокера, в которую начнётся работа с этим событием), repeat (раз в сколько секунд отправка события повторяется, если -1, то событие одноразовое)

# Контракт

От брокера ожидается JSON из массива, каждый элемент которого содержит:

- query (string) -- какой запрос делают люди -- очевидная необходимость
- delta (integer) -- сколько раз этот запрос уже был отправлен -- считаем, что брокер, исходя из своей внутренней логики, может захотеть группировать запросы
- timestamp (string, date-time format, необязательное поле) -- время отправки запроса -- считаем, что у брокера будет более точная информация об актуальности запроса, но если брокер исходя из своей внутренней логики НЕ отправляет нам timestamp, то мы ставим time.Now(), погрешность в рамках пяти минут должна быть небольшая

# Детали реализации

Сервис делится на 3 подсервиса:

- consumer, отвечает за приём логов от брокера
- handler, отвечает за приём запросов на выдачу topN
- looper, отвечает за внутреннюю ежесекундную обработку данных

Сервис строится на трёх общих хранилищах (примечание: к сожалению, я не успел в рамках этого тестового задания перейти на полноценные базы данных, однако теоретически понятно как это делать: подменить хранилища на базы данных в [commonStructures](liveTop/internal/joint/commonStructures.go), по необходимости расширить Wrap структур геттерами и сеттерами):

- eventQueue, содержит историю событий для удобства динамического удаления по истечении 5 минут (строгий порядок не гарантирован, но небольшая погрешность удаления в несколько секунд не будет критической)
- queriesMap, содержит динамическое соответствие текстового поискового запроса и количества его использования
- queriesSorted, содержит актуальный в пределах одной секунды топ всех запросов

**consumer** располагает текстом запроса и delta (на которую его необходимо изменить), поэтому сразу вносит изменения в **queriesMap**, а далее добавляет текст запроса, delta и timestamp (полученный из брокера, а при его отсутствии time.Now()) в **eventQueue**

**handler** имеет доступ к **queriesSorted** и просто берёт его срез

**looper** -- основная часть сервиса, он отрабатывает раз в секунду и делает две вещи:

- Удаляет первый элемент **eventQueue**, если для них прошло требуемое время (по умолчанию 5 минут), при удалении сохранённая delta вычитается из **queriesMap**
- Заново строит **queriesSorted** на основе **queriesMap**

Таким образом:

- обработка запроса брокера длиной в m событий происходит за O(m)
- обработка запроса на topN происходит за O(N)
- очистка истёкших данных с помощью **looper** происходит за O\*(1) для одного события брокера
- ежесекундная сортировка запросов с помощью **lopper** происходит за O(n log n), где n -- количество хранящихся в **queriesMap** ключей

Я считаю это оптимальными асимптотиками, так как из условия задачи ясно, что чтение происходит в разы чаще записей, и мы обеспечили чтению лучшую асимптотику.

Проведём немного математики по самому трудозатратному пункту (сортировке): информацию найти тяжело, но возьмём весьма негативную оценку в 100млн поисковых запросов в день, то есть около 350 000 запросов в 5 минут. n log n от этого числа -- около 2млн операций за одну итерацию **looper** (то есть раз в секунду). С этим вполне способен справиться даже домашний компьютер, не говоря о серверах.

# Обоснование архитектуры

Очередь -- очевидный выбор для проверки на удаление данных (если мы можем хотя бы приблизительно сохранять события упорядоченно по времени, то достаточно проверять самые старые и инвертировать их эффект)

Хэш-таблица идеально подходит для основного хранилища, ведь мы должны сделать соответствие string(query)->int(value), причём порядок не важен (в моей реализации), главное быстро делать операции и прибавления, и уменьшения значений по ключу

Наконец, массив для отсортированных данных также логичен, ведь мы его не изменяем (**looper** всегда пересобирает его с нуля), и он позволяет нам делать быстрый слайс и мгновеннно получать ответ при чтении

И, повторюсь, система в целом особенно хороша, так как чтение происходит гораздо чаще записи

Стратегия с бесконечным циклом выбрана, потому что на практике нам абсолютно не важна погрешность в 1 секунду при показе популярных запросов за последние 5 минут. При проблемах реализации можно изменить env переменную LOOP_INTERVAL_SECONDS и значительно снизить нагрузку при сохранении функциональности сервиса

NATS был выбран, так как это один из самых простых вариантов для освоения с нуля (особенно в сравнении с Kafka), а сложная функциональность мне была не нужна

# Компромиссы

Сервис никогда не будет абсолютно точным: погрешности в рамках нескольких секунд не то что допустимы, но вообще неизбежны

Постоянная работа **looper** может большую часть времени «перегонять из пустого в порожнее», никак не используя данные прошлой сортировки, спустя одну секунду после которой могло не так уж много и измениться. Однако учесть это весьма сложно, ведь у нас нет никаких гарантий на то, как именно меняется **queriesMap** между сортировками

# Бизнес-логика

- Неясно, что мы должны делать в случае неверных данных от брокера (например, не соответствующих JSON схеме). Точнее говоря, должны ли мы как-то сообщать брокеру об успешном принятии/жаловаться на ошибки. Так как из логики продукта ожидается огромный поток данных, я считаю, что логика UDP (со стороны брокера отправил-забыл, а со стороны consumer просто дропать ошибки, так как данных итак достаточно) подходит лучше
- Нигде не указаны значения по умолчанию. Я предположил, что для лимита на запрос кому угодно с запасом хватит 100; а N по умолчанию решил вовсе не ставить (то есть упоминание N в запросе обязательно): пользователь не отправляет запросы сам, он лишь открывает виджет на главной странице, а запрос заранее привязан к этому виджету командой фронтэнда, и я уверен, что команда фронтэнда должна отчётливо понимать, сколько именно продуктов им нужно получить для заполнения виджета
- Не совсем ясно, нужна ли дополнительная информация в topN запросе (например не только названия запросов, но и само количество за 5 минут, на основе которого сделана сортировка), решил оставить без дополнительной информации, так как если мы отдаём пользователю какую-то внутреннюю информацию, внутренние статистики, то мы должны чётко отдавать себе в этом отчёт и убедиться, что это нельзя будет использовать против нас, то есть в реальности как минимум нужно обсудить это с начальством/командой
- Нет уточнения на поведение при равном количестве поисков разных запросов. В таких случаях можно либо делать сортировку в алфавитном порядке, либо не прописывать отдельных условий (оставить случайность). Я выбрал второй вариант, так как нет нужды в детерминированности выдачи, и я не хочу выделять какие-то запросы исключительно по лексикографическим причинам
